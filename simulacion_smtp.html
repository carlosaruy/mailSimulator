Enfoque educativo

Dos formas muy diferentes de "autenticar"

Esta simulación muestra los dos flujos reales del correo electrónico. El primer escenario usa autenticación formal (usuario + contraseña + TLS). El segundo escenario no usa autenticación con credenciales. La confianza se basa en DNS, IP y registros públicos.

Lo más importante de entender

Entre servidores de correo (MTA ↔ MTA) no se pide usuario ni contraseña. El servidor receptor decide aceptar el mensaje porque la IP + el nombre declarado + los registros SPF/DKIM demuestran que el remitente está autorizado.

Qué está pasando
Los dos servidores no intercambian credenciales. La confianza se basa en DNS (SPF/DKIM/DMARC), la IP de origen y reputación. TLS, DKIM y DMARC se verifican en diferentes momentos del flujo.

Comparación clave: ¿Por qué un caso requiere contraseña y el otro no?

Escenario 1 — MUA a MSA
  • • El cliente es un usuario final (no un servidor)
  • • El servidor exige credenciales reales (AUTH LOGIN)
  • • TLS es obligatorio antes de enviar la contraseña
  • • Una vez autenticado, el servidor le da permiso para enviar correo
Escenario 2 — MTA a MTA
  • • Ambos lados son servidores de correo (MTA)
  • • No se usa AUTH. Nadie envía usuario/contraseña
  • • El receptor valida mediante DNS público (SPF, MX) + IP de origen + reputación
  • • Acepta el correo solo si el destinatario es local o hay política que lo permite
Este es el modelo real que usa internet para entregar cientos de miles de millones de correos por día sin que todos los servidores tengan credenciales de todos los demás.
